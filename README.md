  ## [Project: pig-tax] 개발 일지 (26.01.05 ~ 01.10)

### 1. 인프라 구축 및 수익화 설계 (01.05 ~ 01.06)

단순한 토이 프로젝트가 아니라 **실제 작동하는 비즈니스 모델**을 구축하는 데 집중한 시기입니다.

- **Backend & DB:** Supabase 연동을 통해 유저 데이터와 서비스 로직의 기반을 마련했습니다.
- **Revenue:** Lemon Squeezy를 연결하여 결제 시스템을 즉시 도입했습니다.
- **UI/UX Foundation:** framer-motion을 도입해 동적인 사용자 경험을 설계하고, 한국어 패치를 통해 접근성을 높였습니다.

### 2. 무한 수정의 늪, 그리고 최적화 (01.07)

서비스의 완성도는 한 끗 차이임을 보여주는 단계입니다.

- **Interface Improvement:** 이미지 리사이징, 위치 조정, 배경 투명도 조절 등 **시각적 완성도**를 높이는 작업을 반복했습니다.
- **Debugging:** API 오류와 타이핑 최소 입력 문제를 해결하며 서비스 안정성을 확보했습니다.

### 3. 전환율을 높이는 '랜딩 & 성과' 페이지 (01.08 ~ 01.09)

사용자가 들어와서 결제하고 나가는 '유입 경로'를 다듬었습니다.

- **Landing Page:** 서비스의 얼굴인 랜딩 페이지와 환불 규정(refund_info)을 추가해 신뢰도를 높였습니다.
- **Communication:** 이메일 발송 기능 ( resend API ) 을 추가하고 수정하여, 사용자가 서비스 이용 후 즉각적인 피드백을 받을 수 있도록 설계했습니다.

### 4. 운영 자동화 및 관리자 기능 (Phase 4: 01.10 ~ Present)

내가 직접 움직이지 않아도 돌아가는 '시스템'을 만드는 단계입니다.

- **Admin System:** 관리자 페이지에서 데이터를 추가/삭제/숨김 처리할 수 있는 기능을 구현했습니다.
- **Notification Automation:** 결과에 따른 이메일 발송 로직(success_or_not_email_add)을 완성하여 운영 공수를 줄였습니다.
- **DB Optimization:** 데이터베이스 ( Supabase ) 컬럼을 업데이트하며 확장성을 고려한 구조로 개선했습니다.

- 승인 거부 메일을 받은 이후, 나의 서비스를 점검해보았다. 돈을 예치받아서 미션을 성공하면 준다는 모델인 점, TAX라는 용어를 사용한 점, 제 3자의 성공 여부 확인으로 인한 법적 분쟁 가능성 등등… 생각 못한 요인들에서 승인 거부 가능성을 확인하게 되었다.

- 대충 만들고 일단 던져봐서, 이 정도의 타격인 것이라 생각한다. 만약 정말 애지중지해서 겨우 출시했는데, 이런 상상도 못한 벽이 있었다? 그걸로 인해 그 복잡한 배선과 같은 코드를 다시 뜯어고친다? 매우 낙담했을 것이다.
- ‘결제 대행, 결제’ 부분이 정말 크다는 것을 느끼게 되었다. 아무리 잘 만들어도 결제를 담당해주는 플랫폼이나 기관이 막아버리면 시작도 못한다. 개발과 마케팅만이 아니라 이것 또한 새로운 판단 조건이 되었다.
- "이 아이디어는 금융 규제 때문에 결제가 불가능하다"라는 **가장 비싼 데이터**를 최소한의 비용(시간과 에너지)으로 사온 것이다. 애지중지하지 않아서 다행이다. 이것이 MVP의 역할인 것 같다. 만약 6개월 동안 자본을 투입하여 완벽한 코드를 짜고, 아름다운 디자인을 입힌 뒤에 이 결과를 받았다면 그 타격은 회복 불가능했을지도 모른다.

  # [ 고찰 ]
  
  그리고 여기부터는 저의 개인적인 기록입니다.
  
  이 프로젝트를 시작하게 된 계기는 ‘돈’이었습니다.
  
  단지 속물적이다, 돈 밝힌다의 차원이 아니라, 결국 돈이 들어오는 서비스를 직접 해봐야 개발과 마케팅도 그 만큼 노력을 들이게 된다고 느꼈기 때문입니다.
  
  단순히 공모전만 생각해도 그렇습니다. 만약 상금이 3000원인데 쏟아야 하는 시간과 노력이 크다면 누가 열심히 할까요?
  
  저는 세상을 커다란 공모전이라 생각해보기로 했습니다. 다만, 기관이 아니라 제가 직접 기회를 찾는 것이죠
  
  시장이라는 곳은 냉정합니다. 노력이니 사정 따위니 봐주지 않습니다. 단지 ‘자신에게 가치를 주었는가?’ 만을 볼 뿐이죠. 가치를 주지 않았다면 돈을 1원도 내지 않습니다. 돈은 인간의 시간과 생명력과 맞바꾸는 자원이니까요. 당연한 것이죠.
  
  사실 지금의 저는 개발자라기 보다는 ‘실험가’에 가깝습니다. 다만 좀 오래 잡고 있을 수 있는 것 뿐이죠.
  
  처음에는 이러한 방식에 회의감이 많았습니다. ‘나에게 쌓이는 실력이 없는데?’라는 생각에 말이죠. 그렇지만, 삶을 살아가며 돈이라는 것의 냉혹함을 점점 더 경험하고 목격할 수록 그런 배부른 소리를 하기는 어렵다는 생각을 하게 되었습니다. 물론 알고리즘, 자료구조, 영어 능력, 코드를 가독성 있게 짜는 것 등 개발자에게 필요한 역량은 절대적입니다. 더 높이 올라가기 위해서 말이죠.
  
  그럼에도, 저는 시장을 먼저 찔러서 파동을 일으키고 싶다는 욕망이 늘 있었습니다. 이 E_Cycle. 1, Pig-Tax 는 커다란 해외 시장에서 처음으로 그 욕망을 발현시킨 경험인 것이죠.
  
  이러한 시도가 당장에는 의미가 있는 지는 잘 모릅니다. 그럼에도 멈추지는 못합니다. 왜냐하면 이 방향이 맞다고 생각하기 때문입니다. 당장은 허접할 지라도, 저는 스스로 시장의 니즈를 파악하는 존재가 되고 싶습니다. 저는 늘 그런 생각을 품고 살아왔고, 점점 현실에서 실행하는 것들이 늘어나고 있습니다.
